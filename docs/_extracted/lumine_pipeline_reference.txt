 Lumine Paper — Pipeline Reference Data Collection, Pre-Training, Fine-Tuning — Distilled from Paper + Discussion 1. What is Lumine? Lumine is a 7B-parameter VLM-based game agent built on Qwen2-VL-7B-Base that plays Genshin Impact. It takes raw pixel frames as input and outputs keyboard + mouse actions as text. It uses a hybrid reasoning strategy — generating inner monologue only when needed, otherwise directly producing actions for real-time control at 5Hz. 2. The 48.5B Token Question The pie chart in Figure 4 shows token counts for Stage I Pre-training, not parameter counts. The 48.5B is the total training tokens. Data Type Tokens Share What it is Genshin PT 37.1B ~76% Raw gameplay video-action pairs, no labels Web Data 10.8B ~22% General multimodal web data to preserve VLM capabilities Genshin IF 0.6B ~2% Small instruction-following slice for evaluation The 37.1B tokens come from iterating over 1731 hours of gameplay across multiple epochs. At 5Hz that is ~31M frames; each produces ~10-15 action tokens, requiring multiple passes to reach 37.1B. 3. Data Collection Stack 3.1 Video Capture OBS records at 1080p, 60fps, 10,000 kbps, saved as .mkv Monitor forced to 1080p + 100% scaling + Seamless Fullscreen across all 70 annotators All unnecessary apps closed to prevent interruption artifacts 3.2 Keyboard + Mouse Logging (Two Simultaneous Systems) System 1 — Win32 SetWindowsHookEx (global hook) Captures keydown/keyup events and absolute mouse cursor position Timestamps via GetSystemTimePreciseAsFileTime (high-res, not GetTickCount) System 2 — DirectX DirectInput API (active polling) Polls relative mouse displacement every 5ms Needed because Genshin silently recenters cursor during overworld — absolute coords alone would record that as fake input Why Both? Neither alone works: Absolute position fails in 3D overworld — game recenters cursor invisibly Relative displacement fails in GUI scenes — Windows Pointer Precision applies nonlinear scaling Solution: use relative for overworld, absolute diff for GUI, switch via template matching on current frame 3.3 Time Alignment Problem OBS and input logger are triggered by the same hotkey but OBS takes 800ms-2 seconds to write the first frame due to encoder init. Fix: parse absolute timestamp of first video frame from OBS log, compare with input log start, discard events before that point. 3.4 Running During Gameplay — Orchestrator A background Python process listens for a hotkey (e.g. F9) at the OS level via SetWindowsHookEx — the game never loses focus. On press: start OBS via WebSocket API + start input logger simultaneously. Use suppress=True on the hotkey to prevent the game from receiving it. Confirmation without on-screen artifacts Sound cues via winsound.Beep — two short beeps on start, one low beep on stop Log file for post-session verification — check timestamps, durations, sync status Avoid toast notifications or OSD overlays — they get captured in the video 4. Post-Processing Pipeline Input: gameplay.mkv + input_log.jsonl → Output: (frame, action_string) pairs Step 1 — Time Alignment VIDEO_START_MS = parse_obs_log() # e.g. 1200ms events = [e for e in events if e['t'] >= VIDEO_START_MS] for e in events: e['t'] -= VIDEO_START_MS # re-zero Step 2 — Frame Sampling at 5Hz 2 min video = 7200 frames at 60fps. At 5Hz, sample every 200ms = 600 total frames. frame_idx = int(t_ms / 1000 * 60) # map ms timestamp to 60fps index Step 3 — Action String Construction Each 200ms window splits into 6 chunks of 33ms. For each chunk, maintain keyboard state by replaying events. Sum relative mouse deltas for the full window. Action format: <|action_start|> DX DY DZ ; K1 ; K2 ; K3 ; K4 ; K5 ; K6 <|action_end|> Concrete example for t=0 to t=200ms: events: keydown w @ 0ms, mouse_rel dx=3 dy=-1 @ 5ms, mouse_rel dx=2 dy=0 @ 10ms keydown lbutton @ 33ms, keyup lbutton @ 66ms, keydown shift @ 100ms dx=0, dy=1, dz=0 chunk1(0-33ms): {w} -> 'w' chunk2(33-66ms): {w,lbutton} -> 'lbutton,w' chunk3(66-99ms): {w} -> 'w' chunk4(99-132ms): {w,shift} -> 'shift,w' chunk5+6: {w,shift} -> 'shift,w' result: '0 1 0 ; w ; lbutton,w ; w ; shift,w ; shift,w ; shift,w' Step 4 — Output Format Final pairs.jsonl (Genshin PT dataset): {"frame": "frames/000000.jpg", "action": "0 1 0 ; w ; lbutton,w ; w ; shift,w ; shift,w ; shift,w"} {"frame": "frames/000001.jpg", "action": "15 0 0 ; w ; w ; w ; ; ; "} 5. Three Data Types — Full Format Type 1 — Genshin PT (no instruction, 37.1B tokens) <|im_start|>user {frame_t} <|im_end|> <|im_start|>assistant <|action_start|> -23 5 0 ; w ; w ; w,lbutton ; w ; ; <|action_end|> <|im_end|> Type 2 — Web Data (generic VLM, 10.8B tokens) <|im_start|>user {image} What is shown in this image? <|im_end|> <|im_start|>assistant The image shows a mountain landscape... <|im_end|> Type 3 — Genshin IF (with instruction, 0.6B tokens) <|im_start|>user {frame_t} Navigate to the waypoint marked on the minimap. <|im_end|> <|im_start|>assistant <|action_start|> 120 -30 0 ; w ; w ; w ; w ; w ; w <|action_end|> <|im_end|> All three types use identical next-token prediction loss on action tokens. The instruction text is just context, not supervised. Same objective, different conditioning. 6. Pre-Training vs Fine-Tuning — The Real Difference Dimension Pre-Training (Stage I) Instruction Following (Stage II) What it learns HOW actions work — motor primitives WHY to do actions — language grounding Input Frame only Frame + text instruction Output Action string Action string Labels None GPT-4.1 generated Data hours 1731h raw gameplay 165h annotated → 200h filtered Tokens 48.5B total 8B total Loss computed on Action tokens Action tokens Analogy — Learning to Drive Stage Driving Equivalent Pre-training Hours in a parking lot — learning clutch, steering, braking feel Instruction following Actual navigation — turn left at the light, park here Reasoning Defensive driving — that car might cut in, slow down now Why not do instruction following from scratch? Data scale: 1731h raw vs only 165h annotated. Human annotation is expensive — you'd throw away 90% of data. Corner cases: raw gameplay captures weird, exploratory, suboptimal behavior. Clean IF data is too perfect — model never learns to recover from mistakes. 7. Three-Stage Training Summary Stage Output Model Data What it learns I — Pre-training Lumine-Base 37.1B Genshin PT + 10.8B Web + 0.6B IF Action primitives from raw imitation II — Instruction Following Lumine-Instruct Genshin IF + web data same ratio Map language goals to learned actions III — Reasoning Lumine-Thinking 15h human-annotated inner monologues Autonomous reasoning for long-horizon tasks Generated from Lumine paper (arxiv) + conversation — for LLM context handoff
